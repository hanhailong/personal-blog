## 引言

如果你在关注 AI 开发，一定频繁听到这几个词：**LLM**、**Agent**、**Skills**、**MCP**。它们经常被放在一起讨论，但各自代表什么？之间是什么关系？这篇文章尝试用一条清晰的线索把它们串起来。

简单来说：**LLM 是大脑，Agent 是身体，Skills 是技能包，MCP 是神经系统。**

## LLM：大脑

**大语言模型（Large Language Model）** 是一切的基础。它本质上是一个在海量文本上训练出来的神经网络，能够理解自然语言并生成回复。

今天的 LLM（如 Claude、GPT、Gemini）已经具备了几个关键能力：

- **文本理解** — 阅读和分析任意文本
- **文本生成** — 写作、翻译、总结、代码生成
- **上下文学习** — 从提示词和示例中即时学习任务模式
- **指令遵循** — 按照自然语言指令调整行为

但 LLM 本身有一个根本局限：它是一颗"缸中之脑"。它无法访问互联网，无法读写文件，无法调用 API，也无法记住上一次对话的内容。它只能接收文本输入，产出文本输出。

```
┌─────────────────────────┐
│         LLM             │
│                         │
│   文本输入 → 神经网络 → 文本输出  │
│                         │
│   无法：                  │
│   · 联网搜索              │
│   · 执行代码              │
│   · 读写文件              │
│   · 记住对话              │
└─────────────────────────┘
```

## Agent：身体

**AI Agent** 是给 LLM 装上"身体"，让它能够与外部世界交互。

一个 Agent 的核心循环是：

1. **感知** — 接收用户输入和环境反馈
2. **思考** — LLM 分析现状，制定计划
3. **行动** — 调用工具执行操作
4. **观察** — 获取结果，评估是否达成目标
5. **循环** — 未完成则回到步骤 2

```
┌────────────────────────────────────────┐
│              AI Agent                  │
│                                        │
│   ┌──────┐     ┌──────┐               │
│   │ LLM  │────▶│ 工具  │               │
│   │ (大脑)│     │ (手脚) │              │
│   └──────┘     └──┬───┘               │
│        ▲           │                    │
│        │     ┌─────▼─────┐             │
│        └─────│ 记忆/状态  │             │
│              └───────────┘             │
│                                        │
│   工具可以包括：                         │
│   · 网络搜索 · 代码执行 · API 调用       │
│   · 文件读写 · 数据库查询 · 邮件发送     │
└────────────────────────────────────────┘
```

关键在于，LLM 不只是生成文本，而是**决定何时调用哪个工具、如何解读工具返回的结果、根据结果调整下一步计划**。这让 Agent 从"一问一答"进化到"自主完成任务"。

## Skills：技能包

当 Agent 的能力越来越多，一个问题自然浮现：如何组织和复用这些能力？

**Skills** 正是对这个问题的回答。一个 Skill 是对特定领域能力的封装，它包含了：

- **领域知识** — 该领域的专业上下文和最佳实践
- **工具配置** — 完成任务所需的工具集合
- **行为指令** — 在什么情况下触发、如何执行、输出什么格式

以 Claude Code 为例，它内置了多种 Skills：

| Skill | 用途 |
|-------|------|
| `code-review` | 审查代码变更，检查 bug 和改进点 |
| `security-review` | 安全审查当前分支的变更 |
| `claude-api` | 构建和调试 Claude API 应用 |
| `loop` | 周期性执行某个命令或提示词 |
| `run` | 启动并驱动项目应用 |
| `verify` | 验证代码变更是否真的生效 |

Skills 的价值在于**专业化**。通用 Agent 能做很多事，但有了 Skill 加持，Agent 在特定领域的表现会有质的提升——就像一个全科医生 vs 一个心脏外科专家，区别在于专业知识的深度。

从架构上看，Skills 像是 Agent 的"预装模块"：

```
┌──────────────────────────────────────┐
│             AI Agent                 │
│                                      │
│  ┌─────────┐  ┌─────────┐           │
│  │ Skill A │  │ Skill B │  ...      │
│  │  代码审查 │  │  安全审查 │           │
│  └────┬────┘  └────┬────┘           │
│       │            │                 │
│       ▼            ▼                 │
│  ┌──────────────────────┐           │
│  │       工具层          │           │
│  │  grep  edit  bash  git  ...      │
│  └──────────┬───────────┘           │
│             │                        │
│             ▼                        │
│  ┌──────────────────────┐           │
│  │        LLM           │           │
│  └──────────────────────┘           │
└──────────────────────────────────────┘
```

## MCP：神经系统

当 Agent 需要连接的工具越来越多、来源越来越分散，又产生了新问题：**如何标准化 Agent 与外部工具的通信？**

**Model Context Protocol（MCP）** 是 Anthropic 推出的开放协议，旨在解决这个问题。你可以把它理解为 AI 世界的"USB-C 接口"——无论什么工具，只要遵循 MCP 协议，就能被任何支持 MCP 的 Agent 调用。

### 为什么需要 MCP？

在 MCP 出现之前，每个 Agent 框架都有自己的工具定义方式：

```python
# 框架 A 的工具定义
tool_a = {
    "type": "function",
    "function": { "name": "search", "parameters": {...} }
}

# 框架 B 的工具定义
tool_b = {
    "action": "call_api",
    "config": { "method": "search", "args": {...} }
}
```

这意味着每个工具集成都是**一次性**的——为 Claude 写的工具无法用于 GPT，反之亦然。MCP 改变了这一点。

### MCP 的架构

MCP 采用客户端-服务器架构：

```
┌──────────────┐    MCP Protocol    ┌──────────────┐
│              │◀──────────────────▶│              │
│  MCP Client  │    JSON-RPC       │  MCP Server  │
│  (Agent)     │                   │  (Tool)      │
│              │                   │              │
└──────────────┘                   └──────┬───────┘
                                          │
                              ┌───────────┼───────────┐
                              │           │           │
                              ▼           ▼           ▼
                         ┌────────┐ ┌────────┐ ┌────────┐
                         │ 数据库  │ │文件系统│ │外部 API│
                         └────────┘ └────────┘ └────────┘
```

一个 MCP Server 暴露三类原语：

- **Resources** — 可读取的数据源（文件、数据库记录、API 响应等）
- **Tools** — 可执行的操作（搜索、写入、计算等）
- **Prompts** — 预定义的提示词模板

MCP Client（即 Agent）通过 JSON-RPC 与 Server 通信，发现可用资源、调用工具、获取结果。

### 一个具体的例子

假设你要让 Agent 能够查询 GitHub Issues：

1. 开发者实现一个 `github-mcp-server`，暴露 `list_issues`、`create_issue` 等工具
2. Agent 配置中注册这个 MCP Server
3. 当用户说"列出所有标记为 bug 的 issue"，Agent 自动调用 `list_issues` 工具
4. 整个过程对用户透明——Agent 自己做决定、自己调用工具、自己解读结果

```json
// MCP Server 暴露的工具定义（简化）
{
  "name": "list_issues",
  "description": "列出 GitHub 仓库的 issues",
  "inputSchema": {
    "type": "object",
    "properties": {
      "owner": { "type": "string" },
      "repo": { "type": "string" },
      "labels": { "type": "array", "items": { "type": "string" } }
    }
  }
}
```

### MCP 生态

目前 MCP 生态正在快速发展：

- **官方 SDK** — Python、TypeScript、Kotlin 三种语言的 SDK
- **预置 Server** — Google Drive、Slack、GitHub、PostgreSQL 等常见服务的 MCP Server 可直接使用
- **工具链** — Claude Desktop、Claude Code、VS Code 扩展均已支持 MCP

## 从趋势看未来

这些概念不是孤立的发明，而是一条演进路径上的节点：

1. **2022-2023** — LLM 爆发，大家惊叹于模型本身的文字能力
2. **2024** — Agent 兴起，人们发现给模型装上工具后它能做更多
3. **2025** — MCP 和 A2A 协议出现，标准化让 Agent 生态从碎片化走向互联互通
4. **2026** — Skills 和专业化 Agent 成为主流，从"一个通用 Agent 做所有事"转向"多个专业 Agent 协作"

这个趋势指向的方向很明确：**AI 正在从工具变成平台**。就像操作系统提供了文件系统、进程管理、网络栈等基础能力，MCP + Skills 正在成为 AI 应用的"操作系统层"——LLM 负责计算，Agent 负责调度，Skills 是应用，MCP 是通信总线。

## 结语

理解 LLM、Agent、Skills、MCP 这四个概念的关系，是理解当下 AI 开发生态的钥匙。它们各司其职又相互依存：

- **LLM** 提供理解和生成的原始能力
- **Agent** 将这种能力转化为自主完成任务的系统
- **Skills** 让 Agent 在特定领域达到专业水平
- **MCP** 让 Agent 以标准化的方式接入无限的外部世界

如果你正在构建 AI 应用，值得从这四个维度思考架构：选什么模型？Agent 循环怎么设计？哪些能力适合封装成 Skill？用什么 MCP Server 接入外部数据和服务？
